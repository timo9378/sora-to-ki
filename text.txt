
身為一個在程式碼世界裡打滾多年的老兵，我得說，最常讓我們摔跤的，往往不是什麼高深的演算法，而是那些看似不起眼的「環境設定」。今天，就來聊聊一個我在 Vite 專案中踩過的坑，以及如何從中找到最適合自己的解決方案。

### 故事的開端：一個不斷修改的 `proxy`

我的專案是個標準的前後端分離架構，前端用 Vite + React，後端跑在另一個服務上。為了方便本地開發，我需要在 `vite.config.js` 裡設定一個代理 (proxy)，將 `/api` 的請求轉發到後端伺服器。

在本地開發時，我的後端跑在 `http://localhost:3001`。但當我要用 Docker 建構成生產環境映像檔時，前端容器需要透過 Docker 的內部網路，連到名為 `backend` 的後端容器，地址就變成了 `http://backend:3001`。

這就導致了一個令人抓狂的循環：

1.  執行 `npm run dev` 前，我得確保 `target` 是 `localhost`。
2.  執行 `npm run build` 打包 Docker 映像檔前，我又得手動把 `target` 改成 `backend`。

一次兩次還好，但日子一久，忘記修改導致的建構失敗、或是上線後 API 404 的問題，開始頻繁地消磨我的耐心。我知道，一定有更好的方法。

### 探索之路一：業界的黃金標準 `.env`

稍微有點經驗的工程師，第一個想到的絕對是「環境變數」。Vite 對此有著絕佳的原生支援，透過 `.env` 檔案，我們可以將設定與程式碼徹底分離。

這條路被譽為「黃金標準」是有原因的：它清晰、可擴展，而且超級乾淨。

**第一步：建立環境檔案**

在專案根目錄下，我建立了兩個檔案：

`_1. .env.development_` (給 `npm run dev` 使用)
```sh
# 本地開發時，API 伺服器的目標
VITE_API_PROXY_TARGET=http://localhost:3001
```

`_2. .env.production_` (給 `npm run build` 使用)
```sh
# 生產環境構建時 (在 Docker 中)，API 伺服器的目標
VITE_API_PROXY_TARGET=http://backend:3001
```

> **前輩的提醒**：在 Vite 中，只有以 `VITE_` 開頭的環境變數才會被暴露給前端程式碼，這是一個重要的安全約定。

**第二步：改造 `vite.config.js`**

接著，我需要讓設定檔去讀取這些變數。Vite 貼心地提供了 `loadEnv` 這個函式。

```javascript
// vite.config.js
import { defineConfig, loadEnv } from 'vite'; // 1. 引入 loadEnv

export default defineConfig(({ command, mode }) => { // 2. 接收 mode 參數
  // 3. 根據當前模式載入對應的 .env 檔案
  const env = loadEnv(mode, process.cwd(), '');

  return {
    // ... 其他設定
    server: {
      // ...
      proxy: {
        '/api': {
          // 4. 使用環境變數！再也不用手動改了！
          target: env.VITE_API_PROXY_TARGET,
          changeOrigin: true,
          // ...
        },
      },
    },
  };
});
```

搞定！現在，當我運行 `npm run dev`，Vite 會自動載入 `.env.development`；運行 `npm run build` 時，則會載入 `.env.production`。我的雙手終於自由了！這方案堪稱完美，不是嗎？

### 探索之路二：務實主義者的捷徑 `command`

就在我準備提交程式碼時，我停頓了一下。我的專案目前只有我一個人在維護，環境也只有「本地」和「生產」兩種。為了這個簡單的需求，新增兩個檔案，再引入 `loadEnv` 函式，是不是有點「殺雞用牛刀」了？

我回頭看了一眼 `defineConfig` 的參數：`({ command, mode })`。等等，這個 `command` 不就是我需要的嗎？

-   當執行 `npm run dev` (或 `vite serve`)，`command` 的值是 `'serve'`。
-   當執行 `npm run build` (或 `vite build`)，`command` 的值是 `'build'`。

這不就是一個現成的開關嗎？於是，我發現了另一條更直接的路。

```javascript
// vite.config.js
import { defineConfig } from 'vite';

export default defineConfig(({ command }) => { // 只需要 command 參數
  return {
    // ... 其他設定
    server: {
      // ...
      proxy: {
        '/api': {
          // 用一個三元運算子就搞定了！
          target: command === 'serve' ? 'http://localhost:3001' : 'http://backend:3001',
          changeOrigin: true,
          // ...
        },
      },
    },
  };
});
```

一行程式碼，沒有新檔案，沒有新函式。同樣達成了目標，而且程式碼的意圖一目了然。對於我目前這個階段的專案來說，這真是太香了！

### 頓悟的瞬間：沒有最好的，只有最適合的

最後，我選擇了第二種方案。為什麼？

這並不是說 `.env` 的方案不好。恰恰相反，如果我的專案未來有測試環境 (staging)、預覽環境 (preview)，或者需要管理大量的環境變數（例如 API Key），`.env` 絕對是唯一且正確的選擇。它的可擴展性和維護性是無可取代的。

但在當下，我的需求極其單純。`command` 方案以最小的成本解決了最大的痛點。這就是工程師的「務實主義」：**我們追求的不是教科書上的完美範例，而是在特定情境下的最優解。**

### 重點摘要

讓我們來總結一下這趟旅程的收穫：

*   **`.env` 方案 (黃金標準)**
    *   **優點**：配置與程式碼完全分離，擴展性極強，適合複雜專案與團隊協作，是業界公認的最佳實踐。
    *   **缺點**：需要額外建立設定檔，對於極簡專案來說略顯繁瑣。

*   **`command` 判斷方案 (務實捷徑)**
    *   **優點**：極度簡潔，不需額外檔案，修改直觀，完美適用於只有開發/生產兩種環境的簡單專案。
    *   **缺點**：將配置硬編碼在程式碼中，不易擴展到更多環境。

希望我這次踩坑和思考的過程，能為同樣被環境設定所困的你，帶來一些啟發。記住，工具是為我們服務的，選擇最順手的那一把，然後繼續創造酷東西吧！
